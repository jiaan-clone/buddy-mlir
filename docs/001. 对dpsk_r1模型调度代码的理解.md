# 对 dp r1 调度代码的理解

## buddy-deepseek-r1-main.cpp 代码概念
1. 一个关键的数据结构 MemRef<T,N>
 MemRef<T, N>， 例如  `MemRef<float, 4> kv0 = MemRef<float, 4>({1, HeadNum, MaxTokenLength, HiddenSize}, 0);` 代表定义了一个“4维”的数据结构，其中 '4' ，用来初始化 sizes变量（该变量在MemRef模板类中）`intptr_t sizes[4]` ，sizes[0] = 1 、sizes[1] = HeadNum 、sizes[2] = MaxTokenLength 、sizes[3] = HiddenSize，开辟的空间长度为 1 *x HeadNum x MaxTokenLength x HiddenSize ，然后初始值全为 0 。

2. kv-cache 概念
KV Cache 是大模型在生成过程中，为每一层保存历史 token 的 Key 和 Value 向量，后续生成新 token 时直接复用，避免重复计算全部历史 token。
28个Transformer层，每层保存1个K张量和1个V张量，总计28 × 2 = 56个 KV 张量。
每个K/V 张量中包含如下数据：MemRef<float, 4>({1, HeadNum, MaxTokenLength, HiddenSize}, 0)，HeadNum：head头的数量；MaxTokenLength: 最大 token 数目；HiddenSize：每个token对应的维度。（这里HiddenSize真实的名称应为：HeadDim）

3. logits 解读
Prefill 输出 logits 的形状是：
```text
[batch, sequence, vocabulary]
[1,     1024,     151936]
```
假设有效输 token 是：`x0, x1, x2, ..., xL-1`。因果语言模型的训练目标是：
```text
位置0的输出预测x1
位置1的输出预测x2
...
位置L-1的输出预测下一个token xL
```
也就是：`logits[0][t] ≈ P(x[t+1] | x[0..t])`，P(x[t+1] | x[0..t]) 表示：已知位置 0～t 的所有 token 时，下一个 token x[t+1] 出现的概率。
因此，提示词最后一个有效位置的 logits 才用于预测下一个 token：`int tokenIndex = inputContainerPrefill.getTokenCnt() - 1;`
例如有效 token 数为 10 ，最后一个有效位置 = 10 - 1 = 9，logits[0][9]预测位置10应该出现什么 token 

4. decode 循环解读
 1. 获取下一个 decode token 位置
 2. 调用 mlir decode 接口进行推理计算
 3. 统计总耗时和生成 token 数量
 4. 根据 logits 解析token并打印

## buddy-deepseek-r1-main.cpp 代码教学解析

### 1. 文件职责与总体流程

`buddy-deepseek-r1-main.cpp` 不负责实现 Transformer 内部的矩阵计算，而是一个模型推理调度程序。它的主要职责是：

1. 读取用户文本，并转换为 token ID。
2. 加载模型参数。
3. 为 logits 和 KV Cache 分配内存。
4. 调用 MLIR 生成的 `forward_prefill` 处理完整提示词。
5. 根据 prefill logits 选出第一个生成 token。
6. 将 prefill KV Cache 交给 decode 阶段。
7. 循环调用 `forward_decode`，每次生成一个新 token。
8. 输出生成文本和性能统计。

总体数据流如下：

```text
用户文本
  -> tokenizer
  -> prefill token IDs
  -> forward_prefill
       -> logits -> 选出第一个生成 token
       -> 提示词 KV Cache
  -> 复制 KV Cache 到 decode 容器
  -> forward_decode(token + cachePosition + KV Cache)
       -> 更新 KV Cache
       -> 新 logits -> 选出下一个 token
  -> 循环，直到 EOS 或到达长度上限
```

### 2. KV Cache 概念

Transformer 的每个 Attention 层会根据 token 的隐藏状态计算 Query、Key 和 Value：

```text
Q = X * WQ
K = X * WK
V = X * WV
Attention(Q, K, V) = softmax(Q * K^T / sqrt(d)) * V
```

在自回归生成中，每次 decode 只新增一个 token。历史 token 已经计算好的 K/V 不会改变，因此可以保存并在后续 decode 中直接复用，这组历史 K/V 就是 KV Cache。

```text
无 KV Cache：每生成一个 token，都重新计算所有历史 token
有 KV Cache：只计算新 token 的 Q/K/V，历史 K/V 直接复用
```

当前模型有 28 个 Transformer 层，每层需要一个 K 张量和一个 V 张量，因此 C++ 接口中共有：

```text
28 层 * 2 类数据(K、V) = 56 个 KV 张量
```

概念上可以用下列坐标系理解整个 KV Cache：

```text
KV[layer][K/V][batch][head][token][dimension]
   28      2      1     2     1024      128
```

C++ 代码将前两个维度展开成 `kv0` 至 `kv55`，每个 `MemRef` 内部保留后四个维度：

```text
kv0  = 第0层 K Cache
kv1  = 第0层 V Cache
kv2  = 第1层 K Cache
kv3  = 第1层 V Cache
...
kv54 = 第27层 K Cache
kv55 = 第27层 V Cache
```

每个 K 或 V 张量的形状是 `[1, 2, 1024, 128]`：

| 维度 | 当前值 | 含义 | 决定者 |
| --- | ---: | --- | --- |
| Batch | 1 | 一次处理一条序列 | 推理程序/编译配置 |
| `HeadNum` | 2 | 每层的 KV Head 数量 | 模型 `num_key_value_heads` |
| `MaxTokenLength` | 1024 | 提示词和生成 token 共用的 Cache 容量 | 导入时静态配置 |
| `HiddenSize` | 128 | 每个 Head 的 K/V 向量长度，实际是 `HeadDim` | 模型 Attention 结构 |

一个 FP32 K 或 V 张量占用：

```text
1 * 2 * 1024 * 128 * 4 bytes = 1 MiB
```

56 个张量构成的一整套 FP32 KV Cache 约占 56 MiB。

### 3. `MemRef<T, N>` 关键数据结构

`MemRef<T, N>` 是 Buddy 用于表示 MLIR 多维张量的 C++ 容器和描述符：

```cpp
template <typename T, size_t N>
class MemRef {
  T *allocated;
  T *aligned;
  intptr_t offset;
  intptr_t sizes[N];
  intptr_t strides[N];
};
```

| 字段/模板参数 | 作用 |
| --- | --- |
| `T` | 张量元素类型，例如 `float`、`long long` |
| `N` | 编译期维数(rank)，也决定 `sizes` 和 `strides` 数组长度 |
| `allocated` | 原始分配地址，用于释放内存 |
| `aligned` | 实际数据访问地址，`getData()` 返回该指针 |
| `offset` | 数据起点偏移 |
| `sizes[N]` | 每个维度的尺寸，即 shape |
| `strides[N]` | 每个维度前进一步时跳过的元素数 |

例如：

```cpp
MemRef<float, 4> kv({1, 2, 1024, 128}, 0.0f);
```

表示：

```text
元素类型：float
维数：4
形状：[1, 2, 1024, 128]
总元素数：1 * 2 * 1024 * 128
初始值：0.0f
```

`MemRef` 不是分别为四个维度分配内存，而是一次性分配形状乘积大小的连续空间。对于连续布局，上例的 strides 为：

```text
sizes   = [1,      2,      1024, 128]
strides = [262144, 131072, 128,  1]
```

访问 `kv[b][h][t][d]` 时的一维偏移为：

```text
b * strides[0] + h * strides[1] + t * strides[2] + d
```

还需注意：当前 `MemRef` 的拷贝构造函数是深拷贝，会重新分配内存并复制数据。因此将 `kv0` 等对象放入 `PrefillReturns` 或 `DecodeReturns` 时，结构体中的 `MemRef` 拥有独立的底层缓冲区。

### 4. 代码中的主要数据结构

#### 4.1 关键常量

| 常量 | 值 | 含义 |
| --- | ---: | --- |
| `ParamsSize` | 1777088064 | FP32 参数的元素数，加载时按 `sizeof(float) * ParamsSize` 读取 |
| `MaxVocabSize` | 151936 | 模型 logits 的词表维度 |
| `MaxTokenLength` | 1024 | 本静态编译版本的最大 Cache 长度 |
| `HiddenSize` | 128 | 实际是每个 Attention Head 的维度 `HeadDim` |
| `HeadNum` | 2 | 模型每层的 KV Head 数量 |

`NUM_LAYERS` 在当前文件中被设为 56 且没有实际使用。该模型实际有 28 个 Transformer 层，56 是 K/V 张量数量，因此这个常量的命名容易引起误解。

#### 4.2 `Text<size_t, 2>`

`Text<T, N>` 继承自 `MemRef<T, N>`，在张量描述符之外增加了原始字符串、词表映射和有效 token 数量等信息。

```cpp
Text<size_t, 2> inputContainerPrefill(inputStr);
```

刚构造时主要保存 `inputStr`。执行 `tokenizeDeepSeekR1()` 后，它的 MemRef 部分被设置为：

```text
shape = [1, 1024]
第0维 = batch
第1维 = token位置
有效 token 数 = tokenCnt
剩余位置 = pad token
```

#### 4.3 `PrefillReturns`

`PrefillReturns` 是 `_mlir_ciface_forward_prefill` 的多返回值容器，字段顺序为：

```text
kv0 ... kv55, logits
```

- `kv0` 至 `kv55`：prefill 对提示词计算出的 28 层 K/V Cache。
- `logits`：形状为 `[1, 1024, 151936]`，表示每个输入位置对整个词表的原始评分。

该结构体的类型、字段数量和字段顺序必须与 MLIR C ABI 完全一致。

#### 4.4 `DecodeReturns`

`DecodeReturns` 用于匹配 `_mlir_ciface_forward_decode` 的返回值布局，其中包含：

- `cache_position_out`：MLIR 接口返回的 Cache 位置容器。
- `kv0` 至 `kv55`：decode 阶段持有并持续更新的 KV Cache。
- `ret_dummy0` 至 `ret_dummy26`：静态 Cache 降低后为匹配生成接口保留的辅助位置字段。
- `logits`：形状为 `[1, 1, 151936]`，每次 decode 输出一组词表评分。

`cache_position_out` 和 `ret_dummy*` 对 C++ 业务逻辑不是核心数据，但不能随意删除，否则会破坏结构体与 MLIR 多返回值的 ABI 对应。

#### 4.5 `KVPtrArray`

```cpp
using KVPtrArray = std::array<MemRef<float, 4> *, 56>;
```

`PrefillReturns` 和 `DecodeReturns` 使用了 56 个独立字段。`KVPtrArray` 将这些字段的地址收集到数组中，便于通过循环处理：

```text
ptrs[0]  -> kv0
ptrs[1]  -> kv1
...
ptrs[55] -> kv55
```

该数组只保存指针，不复制 KV 数据，也不包含 logits。

### 5. 局部函数说明

| 函数 | 输入 | 输出/状态变化 | 作用 |
| --- | --- | --- | --- |
| `_mlir_ciface_rtclock()` | 无 | 返回当前时间，单位为秒 | 为 MLIR 生成代码提供 C 符号计时接口 |
| `buildPrefillKVPtrs()` | `PrefillReturns&` | 56 个 KV 字段的指针数组 | 便于统一遍历 prefill KV |
| `buildDecodeKVPtrs()` | `DecodeReturns&` | 56 个 KV 字段的指针数组 | 便于统一遍历 decode KV |
| `getUserInput()` | `std::string&` | 修改字符串 | 从标准输入读取一行用户文本 |
| `printLogLabel()` | 无 | 输出日志前缀 | 格式化日志 |
| `printIterInfo()` | 迭代号、token 字符串、耗时 | 打印 token，累加 `total_time` | 输出每轮生成信息 |
| `tokenizeInput()` | 词表路径、`Text&` | 将文本容器填充为 `[1,1024]` token ID | 执行 DeepSeek-R1 分词并计时 |
| `loadParameters()` | 参数文件路径、`MemRef<float,1>&` | 将二进制 FP32 参数读入缓冲区 | 为模型计算准备权重 |
| `findMaxIndex()` | logits 区间的起止指针 | 最大元素的相对下标 | 执行贪心解码，下标即 token ID |
| `copy_kv_by_cache_position_block()` | prefill KV 指针、decode KV 指针、有效 token 数 | 改写 decode KV | 将提示词有效位置的 KV 从 prefill 复制到 decode |

#### `copy_kv_by_cache_position_block()` 的复制单位

该函数遍历 56 个 K/V 张量和每个张量内的 2 个 KV Head。对每个 Head，只复制前 `copy_len` 个有效 token：

```text
bytes_to_copy = copy_len * 128 * sizeof(float)
copy_len = min(cache_position, 1024)
```

逻辑后置条件是：

```text
decodeKV[k][0][head][0:copy_len][0:128]
  = prefillKV[k][0][head][0:copy_len][0:128]
```

### 6. `main()` 函数分块解析

#### 6.1 输出标题并确定文件路径

```cpp
const std::string title = "DeepSeekR1 Inference Powered by Buddy Compiler";
std::string deepSeekR1Dir = DEEPSEEKR1_EXAMPLE_PATH;
std::string deepSeekR1BuildDir = DEEPSEEKR1_EXAMPLE_BUILD_PATH;
const std::string vocabDir = deepSeekR1Dir + "vocab.txt";
const std::string paramsDir = deepSeekR1BuildDir + "arg0.data";
```

这一块确定两个运行时资源：

- `vocab.txt`：token 字符串和 token ID 的映射。
- `arg0.data`：导入模型时导出的 FP32 权重数据。

#### 6.2 读取用户输入并创建基础容器

```cpp
std::string inputStr;
getUserInput(inputStr);

Text<size_t, 2> outputContainer;
Text<size_t, 2> inputContainerPrefill(inputStr);
MemRef<long long, 2> inputContainerDecode({1, 1}, 0LL);
MemRef<float, 1> ParamsContainer({ParamsSize});
MemRef<long long, 1> cachePosition({1}, 0LL);
MemRef<float, 3> logits_prefill({1, MaxTokenLength, MaxVocabSize});
```

各容器的角色：

| 容器 | 角色 |
| --- | --- |
| `inputContainerPrefill` | 保存原始文本，分词后作为 prefill 的 `[1,1024]` token 输入 |
| `inputContainerDecode` | 保存当前 decode token ID，每轮只有一个 token |
| `outputContainer` | 累积生成的 token ID，最后转回字符串 |
| `ParamsContainer` | 保存整个模型的 FP32 参数 |
| `cachePosition` | 表示当前 token 应写入 KV Cache 的位置 |
| `logits_prefill` | 接收 prefill 的 `[1,1024,151936]` 评分 |

#### 6.3 创建 56 个 KV 缓冲区和 prefill 返回容器

```cpp
auto makeKV = []() {
  return MemRef<float, 4>({1, HeadNum, MaxTokenLength, HiddenSize}, 0);
};

MemRef<float, 4> kv0 = makeKV();
...
MemRef<float, 4> kv55 = makeKV();

PrefillReturns prefillRet = {kv0, ..., kv55, logits_prefill};
```

`makeKV()` 统一创建形状为 `[1,2,1024,128]` 且初始值为 0 的 K/V 张量。`PrefillReturns` 将这些输出缓冲区按 MLIR ABI 要求的顺序组织起来。

由于 `MemRef` 拷贝构造是深拷贝，`prefillRet.kv0` 与外层的 `kv0` 不共享同一块底层数据。MLIR prefill 实际写入的是 `prefillRet` 中的缓冲区。

#### 6.4 分词、加载词表和模型参数

```cpp
tokenizeInput(vocabDir, inputContainerPrefill);
outputContainer.loadVocab(vocabDir);
loadParameters(paramsDir, ParamsContainer);
```

`tokenizeInput()` 将文本转换为 token ID，加入 DeepSeek 对话模板的特殊 token，并用 pad token 填充到 1024。`tokenCnt` 只记录有效 token 数，不包括 padding。

`outputContainer` 加载同一词表，以便最后将生成的 token ID 还原为文本。`loadParameters()` 则将 `arg0.data` 按 FP32 数组读入模型参数容器。

#### 6.5 执行 prefill

```cpp
_mlir_ciface_forward_prefill(
    &prefillRet, &ParamsContainer, &inputContainerPrefill);
```

Prefill 一次处理整个提示词，产生两类输出：

```text
prefillRet.kv0 ... kv55：提示词的每层 K/V Cache
prefillRet.logits：每个输入位置的词表评分
```

这是一个 C ABI 输出参数接口：函数的 C++ 返回类型是 `void`，结果被写入 `prefillRet`。

#### 6.6 根据 prefill logits 选出第一个生成 token

```cpp
int tokenIndex = inputContainerPrefill.getTokenCnt() - 1;
const float *startPtr =
    prefillRet.logits.getData() + tokenIndex * MaxVocabSize;
const float *endPtr = startPtr + MaxVocabSize;
int maxIndex = findMaxIndex(startPtr, endPtr);
```

因果语言模型在位置 `t` 的输出用于预测位置 `t+1` 的 token，因此要取最后一个有效提示词位置：

```text
logits[0][tokenIndex][0:151936]
```

`findMaxIndex()` 返回评分最大的词表下标。词表维度的下标就是 token ID，因此 `maxIndex` 是第一个生成 token 的 ID。这里采用的是贪心解码，而不是 top-k、top-p 或温度采样。

```cpp
std::string tok = inputContainerPrefill.getStr(maxIndex);
inputContainerDecode.getData()[0] = static_cast<long long>(maxIndex);
outputContainer.appendTokenIdx(maxIndex);
```

`getStr()` 只是将 token ID 映射成用于打印的 token 字符串。真正传给第一轮 decode 的是整数 `maxIndex`。

#### 6.7 初始化 decode 容器并交接 KV Cache

```cpp
KVPtrArray prefillPtrs = buildPrefillKVPtrs(prefillRet);
MemRef<float, 3> logits_decode({1, 1, MaxVocabSize});
DecodeReturns decodeRet = { /* ABI要求的所有字段 */ };
KVPtrArray decodePtrs = buildDecodeKVPtrs(decodeRet);

copy_kv_by_cache_position_block(
    prefillPtrs, decodePtrs,
    static_cast<int>(inputContainerPrefill.getTokenCnt()));
```

Prefill 和 decode 使用不同的缓冲区。因此在第一次 decode 前，必须将提示词的有效 K/V 从 `prefillRet` 复制到 `decodeRet`。只复制前 `tokenCnt` 个位置，不复制后面的 padding 位置。

```cpp
cachePosition.getData()[0] = inputContainerPrefill.getTokenCnt();
int generateLen = MaxTokenLength - inputContainerPrefill.getTokenCnt();
```

如果提示词有 `L` 个有效 token，那么 Cache 位置 `[0,L)` 已经被占用，第一个 decode 输入 token 的 K/V 应写到位置 `L`。

#### 6.8 Decode 循环

Decode 循环的每轮可以分为四步。

**第一步：同步 Cache 位置**

```cpp
decodeRet.ret_dummy0.getData()[0] = cachePosition.getData()[0];
...
decodeRet.ret_dummy26.getData()[0] = cachePosition.getData()[0];
```

这些辅助字段是生成 MLIR 接口的一部分。C++ 每轮将它们设为当前 `cachePosition`，以保持输入布局和 Cache 位置一致。

**第二步：调用 decode 模型**

```cpp
_mlir_ciface_forward_decode(
    &decodeRet,
    &ParamsContainer,
    &inputContainerDecode,
    &cachePosition,
    &decodeRet.kv0, ...,
    &decodeRet.kv55);
```

该调用的核心契约是：

```text
输入：当前一个 token + 当前位置 + 历史 KV Cache + 模型参数
输出：更新后的 KV Cache + 下一个 token 的 logits
```

在模型内部，当前 token 的 K/V 被写入 `cachePosition`，当前 token 的 Q 与所有有效历史 K 做匹配，再按注意力权重读取历史 V。

**第三步：从 decode logits 中选择 token**

```cpp
const float *startPtr = decodeRet.logits.getData();
const float *endPtr = startPtr + MaxVocabSize;
maxIndex = findMaxIndex(startPtr, endPtr);
```

Decode logits 形状是 `[1,1,151936]`，因此数据首地址就是当前词表评分的起点。若 `maxIndex == 151643`，表示生成了 EOS，立即结束循环。

**第四步：更新下一轮状态**

```cpp
inputContainerDecode.getData()[0] = maxIndex;
outputContainer.appendTokenIdx(maxIndex);
cachePosition.getData()[0] += 1;
```

- 将新选出的 token ID 设为下一轮 decode 的输入。
- 将 token ID 追加到最终输出容器。
- 将下一次 KV 写入位置加 1。

因此 decode 本质上是一个状态循环：

```text
当前 token + cachePosition + KV Cache
  -> forward_decode
  -> logits + 更新后 KV Cache
  -> 选出新 token
  -> 更新 token 和 cachePosition
  -> 下一轮
```

#### 6.9 输出性能和生成文本

```cpp
const double decodeTokensPerSec =
    decodeSeconds > 0.0 ? decodeTokens / decodeSeconds : 0.0;

std::cout << outputContainer.revertDeepSeekR1() << std::endl;
```

最后一块计算 decode tokens/s，并输出总耗时、prefill 吞吐率、decode 吞吐率、原始输入和最终生成文本。`revertDeepSeekR1()` 将 `outputContainer` 中的 token ID 按词表规则还原为 UTF-8 文本。

### 7. 理解该程序时应牢记的契约

1. `HeadNum=2` 表示每个 K 或 V 张量内部有 2 个 KV Head；每层使用两个 `MemRef` 则是因为需要分别保存 K 和 V，两个数字的含义不同。
2. 56 个 `MemRef` 表示 28 层中的 K/V 两类张量；每个张量的内部形状是 `[batch,head,token,dimension]`。
3. `MaxTokenLength=1024` 是提示词 token 和 decode token 共用的静态 Cache 容量，不是模型本身的理论最大上下文。
4. `PrefillReturns` 和 `DecodeReturns` 的字段顺序是 MLIR C ABI 的一部分，不能仅根据 C++ 侧是否“有用”删除字段。
5. `getStr(maxIndex)` 不负责预测 token；`findMaxIndex()` 已经完成 token ID 选择，`getStr()` 只负责 ID 到字符串的映射。
6. Prefill 选出的第一个生成 token 此时尚未写入 KV Cache；它要作为第一轮 decode 输入，才会在 `cachePosition` 处产生并写入自己的 K/V。
7. C++ 形状、数据类型、字段顺序必须与导入后的 MLIR 接口一致；只修改 C++ 常量不能改变已编译模型的静态形状。

### 8. 实现层面的注意点

- `tokenizeDeepSeekR1()` 向长度为 1024 的缓冲区写入 token 时缺少超长输入边界检查，超长分词结果存在越界风险。
- `copy_kv_by_cache_position_block()` 假设 56 个源/目标指针全部有效，形状一致且布局连续，函数内没有验证这些前置条件。
- `cache_position` 必须保持在 `[0,1024)` 可写范围内，否则 decode 写 KV 时存在越界风险。
- `prefillTokensPerSec` 使用 `MaxTokenLength` 而不是实际 `tokenCnt` 计算，衡量的是静态张量长度吞吐率，不一定是用户有效 token 吞吐率。
